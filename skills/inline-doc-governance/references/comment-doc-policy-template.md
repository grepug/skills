# Inline Documentation Policy Template

Adapt this template when a repo needs durable rules for code comments and inline documentation. Keep only the sections that match the repo's languages and tooling.

## Inline Documentation Rules

Use comments to explain intent, contracts, ownership, side effects, edge cases, and non-obvious decisions. Do not add comments that restate the code.

## Coverage Policy

Recommended default:

- Every repo-owned source file should have a short file header before imports or executable code.
- Exported, public, package-visible, cross-boundary, or otherwise important types should have docs immediately above the declaration.
- Public functions and methods should have contract docs when the name and signature do not fully explain caller expectations.
- Inline body comments should explain why a branch, fallback, ordering, timeout, retry, clamp, cache, normalization, assertion, or compatibility workaround exists.

Strict option:

- Every type declaration should have docs, including internal helpers and extensions.

Early adoption option:

- Require file headers first, then add public type and contract docs as code changes.

Recommended choice: use the default policy unless the repo already has a strong reason to require strict all-type docs.

## Generated Files

Never edit generated files or files with generated headers.

Skip generated, vendor-owned, dependency, and build-output files, including:

- generated directories such as `generated/`, `Generated/`, and `__generated__/`
- declaration outputs such as `.d.ts`
- GraphQL generated Swift files such as `*.graphql.swift`
- build outputs such as `dist/`, `build/`, `.build/`, `.next/`, `DerivedData/`, and `Pods/`
- dependency folders such as `node_modules/`
- files whose opening banner says generated, do not edit, do not modify, or generated by

If a generated file has the wrong comments, change the generator or source input.

## Quality Bar

Good comments:

- explain why the code exists or why it is shaped this way
- name important side effects or caller expectations
- describe boundary ownership or public contract meaning
- stay close to the code they describe
- stay short enough to be maintained

Bad comments:

- narrate syntax
- duplicate names, types, or constant values
- teach basic language features
- describe behavior that has drifted from the code
- hide unclear naming that should be fixed instead

## Audit Command

Add the project-specific command here after installing or vendoring an audit script:

```bash
python3 path/to/inline-doc-governance/scripts/audit_inline_docs.py --type-doc-policy public .
```

Use `--type-doc-policy all` only when the repo adopts strict all-type documentation.
